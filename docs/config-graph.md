# Config Graph Direction

Env Mapper's product direction is an agent-safe, read-only configuration
intelligence layer. The current package and repository remain `env-mapper-mcp`
for compatibility, but the long-term product surface is a safe config graph for
AI coding agents and platform teams.

Working naming candidates include `cfggraph`, `config-graph-mcp`,
`agent-config-guard`, and `safe-config-context`. The practical recommendation is
to keep the package name stable through the current OSS milestones and use
`cfggraph` as the graph-first product name when the CLI/API surface is ready for
a deliberate rename.

## Thesis

AI coding agents do not need secret values. They need safe configuration context:

- which env/config keys exist
- where keys are referenced
- which service or runtime target requires them
- whether CI, docs, schemas, or provider metadata mention them
- whether keys are missing, stale, orphaned, dynamic, or unused

Env Mapper is not a secret manager, secret scanner, dotenv linter, or credential
broker. It should become the safe config context layer agents can query instead
of reading `.env` files.

## Security Boundaries

These boundaries are non-negotiable:

- secret values are not read
- secret values are not returned
- secret value hashes are not generated by default
- MCP tools are read-only
- integrations are allowlist-based
- secret-store connectors read key metadata only
- OSS runs local-first
- telemetry is off by default
- write or mutation capability belongs in a separate product surface

Variable names and file paths can still be sensitive metadata. Treat graph
outputs as internal engineering evidence.

## Current Shape

The current implementation is not yet a full graph database, but it already has
the useful graph ingredients:

- config keys: `report.variables[]`
- references/declarations/provider refs: `variable.sources[]`
- findings: `report.findings[]` and `variable.findings[]`
- diff edges: `newFindings`, `resolvedFindings`, `changedClassifications`
- renderers: JSON, Markdown, LLM packet, DMNO draft, dry-run provider plan, SARIF
- delivery channels: CLI, GitHub Action, MCP

Near-term work should evolve this model without breaking the existing report
schema. Add graph-friendly fields behind versioned outputs instead of doing a
large rename or monorepo split prematurely.

## Target Graph Model

Nodes:

- `ConfigKey`
- `CodeReference`
- `Declaration`
- `SecretStoreMetadata`
- `DeploymentTarget`
- `Service`
- `DocumentReference`
- `CIReference`
- `Policy`
- `Finding`

Edges:

- `REFERENCES`
- `DECLARES`
- `REQUIRES`
- `PROVIDED_BY`
- `MISSING_IN`
- `UNUSED_IN`
- `DOCUMENTED_IN`
- `DEPLOYED_TO`
- `OWNED_BY`
- `VIOLATES_POLICY`

Every graph output must include:

- `secretValuesRead: false`
- `secretValuesStored: false`
- `secretValuesIncluded: false`
- enough source evidence for human review

## Wedge

The first killer workflow is PR config impact review for AI-generated code:

1. A PR introduces a new env var such as `OPENAI_API_KEY`.
2. The tool detects that the key is newly referenced.
3. The tool checks safe metadata surfaces such as `.env.example`, docs, CI, and
   provider references.
4. The tool reports missing metadata before deployment.
5. An AI coding agent can explain the deployment risk without seeing the secret
   value.

This is stronger than "a better .env checker" because the value is in agent-safe
configuration topology, not secret access.

## Roadmap Implications

Priority order:

1. Safety: canary leak tests, output guards, no code execution, threat model
2. Agent usefulness: compact context, actionable findings, MCP query tools
3. PR workflow: diff mode, GitHub Action, annotations, SARIF, PR comments
4. Multi-source graph: code, env templates, docs, CI, deployment metadata,
   secret-store metadata
5. Commercial layer: multi-repo history, dashboard, audit log, policy packs

Do not start with broad enterprise dashboard work, secret mutation, or twenty
integrations. Keep the OSS wedge sharp: agents can review config risk without
being handed secrets.
